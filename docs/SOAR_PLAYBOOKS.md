# SOAR-Lite playbooks

Phase 16 adds offline-first, approval-governed response orchestration under `/api/soar` and `/soar`. A playbook is declarative JSON with a `start_step`, at most 100 variables, and at most 50 steps. Allowed step types are action, approval, condition, analyst input, delay, notification, case workflow, evidence snapshot, and end. It contains no Python, JavaScript, SQL, command, URL, callable, module path, nested playbook, or external connector.

Variable references are limited to `${trigger.*}`, `${execution.*}`, `${playbook.*}`, `${steps.*}`, `${analyst_input.*}`, and `${constants.*}`. Dunder names, attributes, environment variables, configuration secrets, ORM objects, and filesystem/network resources are inaccessible. Conditions support exact equality, containment, prefix/suffix, membership, ordered numeric comparisons, existence, booleans, and bounded `all`, `any`, and `not`. Evaluation is strict-typed, deterministic, case sensitivity is explicit, missing values are predictable, nesting is at most eight, and each step performs at most 100 comparisons. No regular-expression or expression evaluator exists.

Lifecycle is draft → testing → active, active → disabled, disabled → active, and draft/testing/disabled → archived. Validation is required before testing and activation. Direct draft activation and archived execution are denied. Every meaningful save creates an immutable sequential version with normalized JSON and SHA-256; unchanged definitions are a no-op warning. Comparison produces a bounded normalized diff. Version rollback copies an old definition into a new version. An in-flight execution retains its exact version.

Execution modes are `dry_run`, `simulation`, and `live_local`. Dry run records intentions and makes no local mutation. Simulation may record explicitly configured harmless local work, but all external-response actions remain simulation-only. Live-local dispatches only fixed ThreatScope database actions. The engine stores append-only events and attempts, checks permission and action policy at dispatch/resume/retry, uses idempotency keys, bounds retries to three, expires at 72 hours, and traverses no more than 50 steps. It has no recursion, sleep, worker, queue, Redis, Celery, or scheduler.

Delay and retry waits persist `next_resume_at`; they never block an HTTP request. An authenticated operator with `soar:execute` calls `POST /api/soar/executions/process-due` manually or from an approved local service scheduler. Processing is oldest-first, bounded to 100, repeat-safe, skips terminal/cancelled rows, and resumes each due step once.

Trigger sources are SOC alert, incident case, confirmed detection match, confirmed threat-intelligence match, vulnerability, phishing analysis, document analysis, and manual. Rules use allowed source fields only, default to proposal-only, apply stable fingerprints, cooldown, an hourly proposal cap, deduplication, and recursion prevention. Automatic-local rules are valid only when every action is read-only or explicitly eligible harmless-local.

Cancellation requires a reason, cancels pending approvals/inputs, stops future dispatch and retries, preserves completed mutations, and never silently rolls them back. Evidence snapshots are redacted before bounded storage and hashed. Reports are static self-contained escaped HTML with 41 documented sections, inline CSS only, no script, active link, or remote asset.
